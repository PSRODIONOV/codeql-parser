# test-project-cpp-branches

Тестовый проект на C++ для проверки **поиска ветвей** и их **динамической
инструментации**. Цель — отразить максимально возможное разнообразие
операторов ветвления, которые отслеживает инструментатор
([queries/cpp/probe_points.ql](../../../../queries/cpp/probe_points.ql)).

## Что отслеживает программа

`probe_points.ql` ставит датчик на тело каждой ветви. Фактически (проверено
прогоном на этой БД) отслеживаются:

| Оператор        | Что отслеживается             | Отслеживается? |
|-----------------|-------------------------------|:--------------:|
| `if`            | then-ветвь                    | ✅ |
| `else`          | else-ветвь (плоский else)     | ✅ |
| `for`           | тело цикла                    | ✅ |
| `while`         | тело цикла                    | ✅ |
| `do-while`      | тело цикла                    | ✅ |
| `try`           | try-блок                      | ✅ |
| `catch`         | тело обработчика              | ✅ |
| `switch`        | каждая метка `case`/`default` | ✅ |
| range-based `for` | —                           | ❌ |
| тернарный `?:`  | —                             | ❌ |
| `&&` / `\|\|`   | короткое замыкание            | ❌ |
| `goto`          | сам переход — нет; управляющий им `if` — да | ❌ |
| ветви из макросов | —                           | ❌ (исключаются) |
| `operator...`, в т.ч. `operator()` лямбд | тело функции/замыкания | ❌ (исключаются по имени) |

`else if` трактуется как самостоятельный `if` (каждое звено цепочки — отдельная
ветвь), а не как одна общая else-ветвь.

Кроме того, на вход каждой функции с определением ставится датчик входа
(`fo:0`) и выхода (`fo:-1`); функции с именем `operator...` (включая `operator()`
лямбд) пропускаются — поэтому ветви ВНУТРИ лямбд не инструментируются.

| `loop_demo.cpp/.h`    | for, вложенные for, for с break/continue, while, do-while, while+if |
| `exception_demo.cpp/.h` | try/catch, несколько catch, вложенные try, try+цикл+if |
| `single_stmt_demo.cpp/.h` | одиночные операторы в теле ветви **без `{}`** (if/for/while/do) |

| `advanced_demo.cpp/.h` | специфические формы, которые **должны** отслеживаться (пустое тело, `do…while(0)`, `for(;;)`, `for` с запятыми, function-try-block, lambda, рекурсия, шаблон, литерал `'{'` в условии, case/default без пробела перед телом) |
| `negative_demo.cpp/.h` | конструкции, которые **НЕ должны** отслеживаться (range-for, `?:`, `&&`/`\|\|`, goto-сам-по-себе, if из макроса, case-метки из одного макровызова). ⚠️ `switch`/`case` здесь — на самом деле ОТСЛЕЖИВАЕТСЯ (см. ниже) |
| `unsafe_demo.cpp/.h` | опасные конструкции (ПОК) для сигнатурного анализа: `strcpy`/`sprintf` (CWE-120), `system` (CWE-078), `printf` с неконст. строкой (CWE-134). Ветвей нет |
| `macro_demo.cpp/.h` | ФО из макросов: (1) **целиком из макроса** (`get_answer`/`get_zero` через `##`) — исключается из всех отчётов и не инструментируется; (2) **только тело/`{` из макроса**, имя настоящее (`macro_body`, паттерн JNI_ENTRY) — инструментируется с учётом макро-`{` (+1 ветвь `if`); (3) **всё тело — один макрос**, имя настоящее (`void macro_full_body() MACRO_BODY;`) — легитимный ФО в статике, но БЕЗ датчика (нет литеральной `{`, тело общее для всех вызовов макроса); (4) **самодостаточный макрос** (`MAKE_ADDER(NAME)`, паттерн `JAVA_INTEGER_OP`) — имя ФО передаётся аргументом (не через `##`), функция легитимна в статике, но датчик молча пропускается по той же причине, что и (3); (5) **макрос-аргумент вызова, сам закрывающий список аргументов** (`check_macro_guard`, паттерн HotSpot `CHECK`/`CHECK_`/`RETURN`/`TRAPS`) — сам `if` НЕ макро-сгенерирован и остаётся в статике, но его тело (вызов, заканчивающийся на `CHECK`) не получает датчика — надёжного места нет |
| `pipeline.cpp/.h`     | класс с методами, вызывающими друг друга — для проверки **маршрутов вызовов** |
| `main.cpp`            | вызывает всё так, чтобы каждая ветвь исполнилась ≥1 раз |

### negative_demo.cpp (регрессионные сторожа)
| ФО                | Конструкция               | Ожидаемо ветвей |
|-------------------|---------------------------|:---------------:|
| `weekday_kind`    | switch/case (+fallthrough)| **8** — ОТСЛЕЖИВАЕТСЯ (7 `case` + 1 `default`) |
| `sum_range`       | range-based for           | 0 ✅ не отслеживается |
| `sign_and_flags`  | `?:`, `&&`, `\|\|`         | 0 ✅ не отслеживается |
| `retry_goto`      | goto                      | 1 — управляющий `if` отслеживается, сам `goto` нет |
| `macro_control`   | `if` из макроса           | 0 ✅ исключается |
| `macro_generated_cases` | case-метки из ОДНОГО макровызова (паттерн `REP8`/`REP16`) + 1 обычная метка + `default` | **2** — макро-метки (4 шт.) исключены целиком, отслеживаются только `case 20` и `default` |

⚠️ `switch`/`case` фактически **инструментируется** (датчик на каждую метку
`case`/`default`) — это НЕ негативный случай. Истинные негативы здесь:
range-based for, `?:`, `&&`/`\|\|`, `if` из макроса, case-метки из одного
макровызова (`CASES4`). В `retry_goto` сам `goto` ветвью не является, но
управляющий им `if` отслеживается.

### Итог (проверено на БД)

Статический `Перечень_ветвей.csv` для этого проекта содержит **104 ветви** в
69 ФО; инструментатор размещает **246 датчиков** (вход/выход + ветви) — 4
ФО (`macro_full_body`, `add_via_macro`, `add_via_macro2`, `constexpr_square`)
легитимны в статике, но без датчика (самодостаточный макрос/constexpr —
нет надёжного места для вставки или вставка ломает компиляцию), плюс
1 ветвь (`check_macro_guard`, if с телом на HotSpot-идиоме CHECK) — ФО
инструментирован (вход/выход есть), но именно эта ветвь без датчика
(см. `core/codeql_analyzer.py`/`dynamic/instrument_c_make.py`),
**0 пропусков** среди прочих, синтаксис инструментированного кода — OK.
Сюда входят `if`/`else`/`for`/`while`/`do`/`try`/`catch`/`case`/`default`.
Ветви внутри лямбды (`count_positive`) НЕ отслеживаются — `operator()`
замыкания исключён по имени.

Покрытые формы записи тела ветви (все инструментируются корректно):
- многострочное тело в `{ }`;
- одиночный оператор без `{ }` (тело на отдельной строке) — `single_stmt_demo`;
- заголовок и тело на одной строке, с `{ }` и без — `oneline_demo`;
- `if/else` в одну строку во всех комбинациях скобок — `oneline_demo`;
- пустое тело (`while (*p++) ;`, `for (…) ;`, `if (…) ; else …`) — `advanced_demo`.

## Маршруты вызовов (для отчёта по функциям/процедурам)

- `Pipeline::process` → `Pipeline::classify` → `Pipeline::above_threshold`
- `Pipeline::process` → `Pipeline::normalize`
- `main` → все функции (см. `main.cpp`)

## Сборка

CMake:
```sh
cmake -S . -B build && cmake --build build
./build/branches
```

Make / напрямую g++:
```sh
make
./branches
```

Все ветви в `main.cpp` вызываются хотя бы один раз, поэтому при сопоставлении
динамических трасс со статическими маршрутами все ветви должны быть отмечены
как исполнявшиеся (`Исполнялся = да`), а непредусмотренных маршрутов быть
не должно.

## Регрессия инструментатора (найденные и исправленные баги)

Фикстуры ниже воспроизводят 9 конкретных багов `instrument_c_make.py`/
`instrument_cpp.py`/`core/codeql_analyzer.py`/`queries/cpp/*.ql`, найденных на
реальных проектах (HotSpot/gosjava) и закреплённых здесь как регресс
(`tests/test_cpp.py`, классы `TestInstrumentorRegressionBugs` — побайтовая
проверка вставки, плюс точечные проверки в `TestBranchDetection`/`TestMacroFO`/
`TestSensorMap`/`TestCoverage`):

| № | Баг | Фикстура | Прототип в реальном коде |
|---|-----|----------|---------------------------|
| 1 | Наивный поиск `{` с начала строки находил литерал `'{'` раньше настоящей открывающей скобки и резал код пополам внутри литерала | `brace_literal_guard` (`advanced_demo.cpp`) | `adlc/adlparse.cpp::get_oplist` (HotSpot) |
| 2 | Самодостаточный макрос (сигнатура+тело+`{}` целиком в макроопределении, имя — аргумент макроса) оборачивался в `{ датчик; ВЫЗОВ }`, ломая компиляцию | `add_via_macro`/`add_via_macro2`, `macro_full_body` (`macro_demo.cpp`) | `JAVA_INTEGER_OP` (`globalDefinitions.hpp`, HotSpot) |
| 3 | Off-by-one в обработчике `"direct"` (case/default) — без пробела перед телом разрезал первое слово (`r`+датчик+`eturn`) | `case_no_space_kind` (`advanced_demo.cpp`) | `c1_LIR.hpp::as_BasicType` (HotSpot) |
| 4 | Утечка `col_shifts` между несколькими `open`/`close` на одной строке — сдвиг от ОДНОЙ вставки переносился на НЕ связанную с ней, разрезая уже вставленный `__TRACE()` | `if_else_oneline_nn` (`oneline_demo.cpp`, уже существовал) | — |
| 5 | Одна метка `case MACRO(x):` разворачивается макросом в несколько `case` — все указывают на одно место, несколько датчиков ломали текст вокруг общего `:` | `macro_generated_cases`/`CASES4` (`negative_demo.cpp`) | `REP8`/`REP16` (`assembler_x86.cpp`, HotSpot) |
| 6 | `sensor_map`/`Карта_датчиков.csv` заполнялся ДО того, как стало известно, удастся ли вставить датчик (баг 2/X-macro) — карта лгала о датчиках, которых в коде нет, и портила Покрытие_ФО (ложное «не покрыт» вместо «не инстр.») | те же ФО, что в баге 2 | — (найдено при добавлении регресса для бага 2) |
| 7 | Макрос — последний аргумент вызова, САМ закрывающий список аргументов и порождающий `if` внутри своего раскрытия (`f(args, CHECK)` -> `f(args, THREAD); if (...) return; ...)`). CodeQL репортует конец одиночного оператора-тела `if` (hasBlock=0) сразу после "CHECK" — ВНУТРИ списка аргументов вызова, а не после настоящего `');'`. Обёртка `{ датчик; ВЫЗОВ }` по такой координате вставляла `}` внутрь аргументов | `check_macro_guard` (`macro_demo.cpp`) | `CHECK`/`CHECK_`/`RETURN`/`TRAPS` (`classFileParser.cpp::classfile_parse_error`, HotSpot) |
| 8 | `while(...) try {...} catch(...) {...}` без своих `{}` вокруг while (тело while — сам `TryStmt`). CodeQL даёт `TryStmt.getLocation()` только до конца try-блока, БЕЗ catch-обработчика, поэтому закрывающая `}` обёртки одиночного оператора (hasBlock=0) попадала ПРЯМО ПЕРЕД `catch` — он оставался "осиротевшим" вне фигурных скобок while, сборка ломалась (`expected 'catch' before '}' token`). Фикс — `queries/cpp/probe_points.ql::properStmtEnd()` берёт конец ПОСЛЕДНЕГО catch-блока вместо `TryStmt.getLocation()` | `while_try_no_brace` (`exception_demo.cpp`) | `rikdataset.cpp::RIKRasterBand::IReadBlock` (GDAL/RIK) |
| 9 | `constexpr`-функция получала датчик: `__TRACE_FN()`/`__TRACE()` вызывают обычные (не `constexpr`) `__trace_enter()`/`__trace_hit()`, что ломает constexpr-вычислимость функции — любой зависимый `static_assert`/constexpr-контекст/шаблон ломается каскадом вторичных ошибок компиляции по всей библиотеке. Фикс — `queries/cpp/probe_points.ql` добавляет `not f.isConstexpr()` (и симметрично для enclosing-функции во всех branch-клаузах) — ФО легитимен в статике, но БЕЗ датчика, как самодостаточный макрос | `constexpr_square` (`advanced_demo.cpp`) | `fmt::v8::monostate::monostate()`, `fmt::v8::detail::is_constant_evaluated()` (header-only `fmt`, `osm2pgsql/contrib/fmt`) — снесло сборку всего проекта (100+ ошибок) |
