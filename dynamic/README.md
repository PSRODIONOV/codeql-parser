# Динамический анализ (датчики · трассы · покрытие)

Дополняет статический анализ: встраивает **датчики** в копию исходников проекта,
после запуска собирает **трассы** выполнения и строит **отчёты о покрытии** ФО и ветвей.

Нумерация ФО и ветвей берётся из статических отчётов (`Перечень_ФО`, `Перечень_ветвей`),
поэтому динамика совпадает со статикой 1:1 — по составу и по номерам.

---

## Поток данных

```
CodeQL БД (src.zip) ──┐
Перечень_ФО.csv ──────┼─► [ИНСТРУМЕНТАТОР] ─► instrumented-sources
Перечень_ветвей.csv ──┘        │               + Карта_датчиков.csv
                               ▼
              пользователь собирает и запускает
                               ▼
                  $HOME/<lang>-<timestamp>-<pid>.log  (трассы, ротация 100 МБ)
                               ▼
   трассы + Карта_датчиков + Перечень_ФО + Перечень_ветвей
                               ▼
            [РЕПОРТЕР] ─► reports/dynamic/{Покрытие_ФО, Покрытие_ветвей, Сводка_покрытия}
```

## Формат трассы

Каждое срабатывание датчика — строка `ФО:значение`:

| Датчик | Запись |
|---|---|
| Вход в ФО | `[№ ФО]:0` |
| Выход из ФО | `[№ ФО]:-1` |
| Ветвь | `[№ ФО]:[№ ветви #N]` |

Анти‑спам: каждый датчик пишет срабатывания №1, 2, 4, 8, 16… (степени двойки),
поэтому цикл из 10⁶ итераций даёт ~20 строк, а не миллион. Трассы пишутся в
домашний каталог, файлами по 100 МБ с именем `<язык>-<timestamp>-<pid>.log`.

## Структура рабочего каталога (workspace)

GUI и оркестратор создают единую структуру под выбранным рабочим каталогом:

```
<workspace>/
├── orig-sources/            чистая копия исходников проекта
├── instrumented-sources/    копия с датчиками (+ рантайм, + Карта_датчиков.csv)
└── reports/
    ├── static/              статика (Перечень_ФО, Перечень_ветвей, …)
    └── dynamic/             покрытие (Покрытие_ФО, Покрытие_ветвей, Сводка_покрытия)
                             + traces/  (если запуск через оркестратор)
```

## Компоненты (каталог `dynamic/`)

| Файл | Назначение |
|---|---|
| `instrument_cpp.py` | C/C++ малые плоские проекты (RAII/cleanup, `g++ -fsyntax-only`) |
| `instrument_c_make.py` | C/C++ со своей сборкой (make и т.п.); single‑header рантайм |
| `instrument_py.py` | Python (декоратор + вызовы, `ast.parse`) |
| `instrument_js.py` | JavaScript / Node, CommonJS (try/finally, `node --check`) |
| `instrument_java.py` | Java (try/finally, `javac`); для Maven — без javac‑проверки |
| `coverage_report.py` | строит 3 отчёта о покрытии (языко‑независимый) |
| `route_match_report.py` | сопоставляет ФАКТИЧЕСКИЕ маршруты (из трасс) со СТАТИЧЕСКИМИ → `Сопоставление_маршрутов.csv` |
| `analyze_project.py` | оркестратор: извлечь src.zip → статика → инструментация → сборка/запуск → покрытие |
| `runtime/` | рантаймы датчиков: `__trace.h`/`__trace_rt.c.tmpl` (C/C++), `__trace_singlehdr.h` (C/C++ со сборкой), `cqtrace.py`, `cqtrace.js`, `Cqtrace.java.tmpl` |
| `drivers/` | примеры драйверов для запуска инструментированных проектов (requests, express, H2) |

**Принцип:** весь рантайм кладётся ВНУТРЬ инструментированной копии — ничего в систему
(`/usr/include`, `/usr/lib`, глобальный classpath). Так инструментация: без админ‑прав,
кроссплатформенна, воспроизводима, легко удаляется (снёс `instrumented-sources`).

---

## Полный цикл

1. **Статика** → `reports/static` (обязательно с `Перечень_ветвей.csv`).
2. **Инструментация**: `orig-sources` → `instrumented-sources` + `Карта_датчиков.csv`.
3. **Сборка и запуск** инструментированного кода (делает пользователь) → трассы в `$HOME`.
4. **Покрытие**: трассы + статика + карта датчиков → `reports/dynamic`.

---

## Варианты инструментации по типам проектов

Инструментатор выбирается по языку и признаку «своя сборка»:

| Язык | Своя сборка | Скрипт | Проверка синтаксиса |
|---|---|---|---|
| C++ | нет | `instrument_cpp.py` | `g++ -fsyntax-only` |
| C++ | да | `instrument_c_make.py` | нет (проверит сборка) |
| Java | нет | `instrument_java.py` | `javac` |
| Java | да | `instrument_java.py --no-syntax-check` | нет |
| Python | — | `instrument_py.py` | `ast.parse` |
| JavaScript | — | `instrument_js.py` | `node --check` |

### 1. Малый C/C++ (плоский)
Вход/выход ФО — через RAII (C++) или `__attribute__((cleanup))` (C). Сборка потом:
`g++ -std=c++14 -pthread -I. *.cpp`.

### 2. C/C++ со своей сборкой (make), напр. nginx, gosjava
Single‑header рантайм: тело датчика целиком в `__trace.h` (weak‑символы — линкер сводит
N копий из N единиц трансляции в одну на конечный бинарник). Скрипт пишет файл в
`<out>/__trace.h`; его нужно вручную скопировать в системный include‑путь (обычно
`/usr/include`) и при необходимости поправить сборочные скрипты/команды компиляции —
никакие `--header-dir`/`--impl-file` не нужны. Сборка: `make`.

### 3. Python, напр. requests
Декоратор `@cqtrace.fn(ФО)` над `def` + `cqtrace._t(ФО, #N)` в теле ветви.
Запуск: ваш драйвер с `PYTHONPATH=<instrumented>;<instrumented>/src`.

### 4. JavaScript (Node), напр. Express
Обёртка тела `{ cqtrace.hit(fo,0); try {…} finally { cqtrace.hit(fo,-1); } }`.
Потом: `npm install` в `instrumented-sources`, затем `node` ваш драйвер.

### 5. Java малый
`Cqtrace.java` в каталоге пакета; проверка `javac` всех файлов.

### 6. Java с Maven/Gradle, напр. H2
Обязательны `--cqtrace-package` и `--cqtrace-dir` (см. ниже). Сборка: `./mvnw -DskipTests compile`.

## Доп. аргументы инструментаторов (подробно)

`--project`, `--db`, `--reports`, `--out`, `--lang` подставляет GUI/оркестратор.
Остальное — по сценарию.

### `--codeql <путь>`
Путь к исполняемому CodeQL для запроса точек вставки (`probe_points.ql`).
**По умолчанию** имя `codeql` резолвится автоматически: локальный
`third-party/codeql-win/codeql.exe` (Windows) или `third-party/codeql-linux/codeql`, затем PATH. Указывать
обычно не нужно. Пример: `--codeql C:/tools/codeql/codeql.exe`.

### C/C++ со своей сборкой — без `--header-dir`/`--impl-file`
Реализация рантайма целиком в заголовке `__trace.h` (функция `__trace_hit` и большие
данные состояния помечены GNU/Clang‑атрибутом `weak`): при подключении из N файлов
линкер сам сводит N копий в ОДИН экземпляр на конечный бинарник — без «multiple
definition» и без выбора одного особого impl‑файла на каждый линкуемый бинарник.
Каждый инструментированный файл получает `#include "__trace.h"`; сам заголовок
скрипт один раз пишет в `<out>/__trace.h`. **Вручную:** скопировать этот файл в
системный include‑путь (`/usr/include`) перед сборкой проекта — тогда `#include
"__trace.h"` находится из любого файла независимо от `-I` путей сборки и от
количества отдельных бинарников (.so/.exe) в проекте.

### `--cqtrace-package <пакет>`  (Java)
Java‑пакет класса рантайма `Cqtrace`; ссылка везде — `<пакет>.Cqtrace.hit(...)`.
**Почему:** автоопределение берёт самый частый пакет — в больших проектах это тестовый
(`org.h2.test.db`), и тогда `Cqtrace` оказывается в `src/test`, а main‑классы его не
видят → ошибки компиляции. Для Maven/Gradle задайте **общий корень пакета**.
**По умолчанию** автоопределение (ок для малых однопакетных). Пример: `--cqtrace-package org.h2`.

### `--cqtrace-dir <каталог>`  (Java)
Куда записать `Cqtrace.java` (относительно копии). Должен соответствовать пакету и
**компилироваться в main‑фазе**.
**Почему:** путь файла обязан совпадать с пакетом (`package org.h2;` → `.../org/h2/`), и
файл должен попасть в `src/main/...`, иначе при `mvn compile` он не скомпилируется.
**По умолчанию** каталог найденного файла с этим пакетом. Пример: `--cqtrace-dir src/main/org/h2`
(вместе с `--cqtrace-package org.h2` создаст `src/main/org/h2/Cqtrace.java`).

### `--no-syntax-check`
Пропустить встроенную проверку синтаксиса. Для проектов со своей сборкой проверка либо
невозможна (нужны все include‑флаги/classpath/зависимости), либо избыточна — её роль
выполняет последующая `make`/`mvn`. **По умолчанию** проверка ВКЛЮЧЕНА. Для **Java** при
включённом чекбоксе «со своей сборкой» флаг добавляется автоматически; для **C++** со
своей сборкой используется `instrument_c_make.py`, который проверку и так не делает.

### Что писать вручную — сводка

| Сценарий | «Доп. аргументы» |
|---|---|
| Малый C++ / Python / JS / малый Java | *(пусто)* |
| C++ со своей сборкой (nginx, gosjava) | *(пусто; скопировать `__trace.h` в `/usr/include` вручную)* |
| Java с Maven (H2) | `--cqtrace-package org.h2 --cqtrace-dir src/main/org/h2` |
| codeql не находится | добавить `--codeql <путь>` |

## Запуск через GUI

1. **▶ Запустить анализ** (статика) → `reports/static`.
2. Группа «🔬 ДИНАМИЧЕСКИЙ АНАЛИЗ»: указать **Исходники:**, при необходимости
   ☑ «со своей сборкой» и **Доп. аргументы** → **🔬 Инструментировать**.
3. Вне GUI собрать и запустить инструментированный код → трассы.
4. Указать **Трассы:** → **📊 Построить покрытие** → `reports/dynamic`.

GUI сам подставляет `--project` (=`orig-sources`), `--db`, `--reports`, `--out`, `--lang`,
а для Java со своей сборкой — ещё `--no-syntax-check`.

## Запуск из командной строки (примеры)

```bash
# 1) статика (нужен Перечень_ветвей)
python main.py <db> -o <ws>/reports/static --language cpp --pattern '%test-project%' --codeql third-party/codeql-linux/codeql

# 2a) малый C++
python dynamic/instrument_cpp.py --project <src> --db <db> \
    --reports <ws>/reports/static --out <ws>/instrumented-sources --lang cpp

# 2b) C++ со своей сборкой (nginx, gosjava)
python dynamic/instrument_c_make.py --project <nginx-tree> --db <db> \
    --reports <ws>/reports/static --out <ws>/instrumented-sources --lang cpp
# затем вручную: cp <ws>/instrumented-sources/__trace.h /usr/include/__trace.h

# 2c) Java с Maven (H2)
python dynamic/instrument_java.py --project <h2-tree> --db <db> \
    --reports <ws>/reports/static --out <ws>/instrumented-sources --lang java \
    --no-syntax-check --cqtrace-package org.h2 --cqtrace-dir src/main/org/h2

# 3) собрать и запустить инструментированный код (зависит от проекта) → трассы в $HOME

# 4) покрытие
python dynamic/coverage_report.py --traces <трассы> \
    --reports <ws>/reports/static --sensor-map <ws>/instrumented-sources/Карта_датчиков.csv \
    --out <ws>/reports/dynamic
```

Полностью автоматический прогон (извлечение src.zip + статика + инструментация + сборка + запуск + покрытие):

```bash
python dynamic/analyze_project.py --name <имя> --db <db> --marker <сегмент-пути> \
    --language cpp --pattern '%...%' --codeql third-party/codeql-linux/codeql --run-build
```

## Отчёты о покрытии

1. **Покрытие_ФО.csv** — `№ ФО; ФО; Покрыт` (да / нет / не инстр.)
2. **Покрытие_ветвей.csv** — Перечень_ветвей + колонка `Покрыта` (да / нет / не инстр.)
3. **Сводка_покрытия.csv** — `№ ФО; ФО; Веток всего; Веток покрыто; % ветвей`

Статусы: **да** — датчик сработал; **нет** — датчик стоит, но не сработал;
**не инстр.** — датчика нет (объект нельзя инструментировать: inline‑функция в
заголовке, системный файл, lambda/comprehension и т.п.). Знаменатель процента —
только инструментированные объекты, чтобы не занижать покрытие неизмеримыми.

## Сопоставление фактических маршрутов со статическими

Покрытие отвечает «какие ветви исполнялись» (множество), но не «какие
**маршруты** реально прошли». Для этого рантайм датчиков ведёт поток буфера
текущего вызова ФО и на выходе выгружает ДВЕ строки (уникальные пути пишутся
1,2,4,8… раз — анти‑спам на уровне маршрута):
- `R fo:b1>b2>...`    — фактический маршрут по сработавшим «да»‑ветвям;
- `C fo:fo>c1>c2>...` — фактическая цепочка вызовов (self, затем вызванные ФО).

Подряд идущие повторы ветви/вызова **схлопываются** прямо в рантайме (итерации
цикла → ветвь один раз; прямая рекурсия `45>45` → `45`), чтобы фактический
маршрут совпадал со статическим базисным (там ветвь цикла указана однократно).
Неподряд идущие повторы (повторный вход после другой ветви) сохраняются.

Это **дополняет** обычные coverage‑строки `fo:br` в той же трассе (старый
`coverage_report.py` их игнорирует).

`route_match_report.py` строит **два** отчёта, зеркально статическим отчётам о
маршрутах:

```bash
python3 route_match_report.py --traces <трассы> --reports <static-dir> --out <dynamic-dir>
```

- **Сопоставление_маршрутов(функций_процедур).csv** — цепочки ВЫЗОВОВ (`C`‑строки vs
  `Маршруты_выполнения_ФО(процедур_функций).csv`). Строится всегда, когда есть
  статический отчёт по вызовам.
- **Сопоставление_маршрутов(ветвей).csv** — последовательности ВЕТВЕЙ (`R`‑строки vs
  `Маршруты_выполнения_ФО(ветвей).csv`). Строится **только если** в статике есть
  отчёт по ветвям (`Перечень_ветвей.csv`) **и** в трассах есть данные о ветвях.

Оба: `№ ФО; ФО; № маршрута; Маршрут; Исполнялся (да/нет); Кол-во; Тип записи`. Тип:
- **статический** — построенный статикой маршрут (отмечается, исполнялся ли);
- **непредусмотренный** — фактический путь, которому нет статического маршрута
  (рекурсия/исключения/`goto` и т.п.);
- **неоднозначно (try)** — только для ветвей: маршруты, различающиеся лишь исходом
  `try` («нет исключения» vs «catch»), по трассе неразличимы (датчик на try‑блоке
  входит в обоих случаях).

В GUI оба отчёта строятся автоматически после «Построить покрытие».

> **Область поддержки.** Запись фактических маршрутов реализована в single‑header
> рантайме (`__trace_singlehdr.h`, т.е. C/C++ «со своей сборкой» / `instrument_c_make`),
> модель однопоточная. Для плоского рантайма (`instrument_cpp`), а также Python/JS/
> Java/PHP — запись маршрутов пока не добавлена (следующий шаг); там отчёт покажет
> статические маршруты со статусом «нет».

## Возможные проблемы

| Симптом | Причина / решение |
|---|---|
| `codeql` не найден | укажите `--codeql <путь>` или положите в `third-party/` (`codeql-win`/`codeql-linux`) |
| ошибка на `[8] Проверка синтаксиса` | нет `g++`/`javac`/`node` в PATH — добавьте в PATH или `--no-syntax-check` |
| `Перечень_ветвей.csv` не найден | сначала статика (с этим отчётом) |
| `node: Cannot find module 'body-parser'` (JS) | `npm install` в `instrumented-sources` |
| трасса пустая, но программа отработала | проверьте, что `HOME` (C/C++/Py/JS) или `-Duser.home` (Java) указывает на каталог трасс |
| Java: main‑классы не видят `Cqtrace` | задайте `--cqtrace-package`/`--cqtrace-dir` в общий корень пакета и `src/main` |
