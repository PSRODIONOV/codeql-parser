# Проектный графический интерфейс (gui_project)

Проектно-ориентированный GUI для статического и динамического анализа кодовых баз
CodeQL. Все сведения проекта хранятся в SQLite (`project.db`), поэтому к проекту
можно возвращаться: отчёты создаются из базы без повторного прогона CodeQL.

## Запуск

```powershell
# рекомендуется (PowerShell)
$env:_JAVA_OPTIONS=""
python run_project_gui.py
```

Требуется **PyQt5** (`pip install -r requirements_gui.txt`). CodeQL, Node.js, Maven,
Pillow поставляются локально (см. README.md → «Развёртывание на новой ЭВМ»).

## Поток работы

### 1. Стартовое окно — выбор проекта
- **➕ Создать проект** — имя + каталог сохранения. Создаётся `<каталог>/<имя>/`
  с `project.db` и структурой `reports/static`, `reports/dynamic/traces`,
  `orig-sources`, `src-instrumented`.
- **📂 Открыть проект** — выбрать существующий `project.db`.
- **История** — нумерованный список недавних проектов (двойной клик — открыть).
  Хранится в `~/.codeql-gui/recent_projects.json`.

### 2. Вкладка «Статический анализ»
1. **Кодовая база CodeQL** — перетащите каталог БД в зону **или** кликните ЛКМ для
   выбора в проводнике. Нужен каталог *базы* (есть `codeql-database.yml`), а не
   исходников.
2. **Язык**, **состав проверок** (1:1 с отчётами).
3. **Фильтры файлов** — белый список (включить) и чёрный список (исключить),
   по одному glob-шаблону на строку (напр. `*/src/*`, `*/tests/*`).
4. **Ограничения** — ОЗУ (шаг 512 МБ) и маршрутов на ФО (шаг 500).
5. **▶ Запустить статический анализ** — данные сохраняются в `project.db`.
   В логе видно время каждого этапа, общий итог и прогресс-бар генерации блок-схем.
6. **📄 Создать отчёты** (активна после анализа) — выгружает CSV + `flowcharts/`
   в `reports/static` из базы.

### 3. Вкладка «Динамический анализ»
Становится активной после выполнения статического анализа.
1. **Исходные тексты для инструментации** — зона drag-and-drop / проводник.
2. **Проект со своей сборкой (make/Maven)** — для C++ переключает на single-header
   рантайм; влияет на подсказки доп. аргументов.
3. **Доп. аргументы** — по чек-боксу; плейсхолдер зависит от языка
   (напр. Java: `--cqtrace-package … --cqtrace-dir …`).
4. **Инструментация ветвей** — доступна, только если в статике собраны данные ветвей.
5. **🔬 Инструментировать** — инструментированные исходники → `src-instrumented/`
   (недостающий «Перечень ФО» создаётся автоматически).
6. **Трассы выполнения** — перетащите `*.log`; каждая трасса увеличивает счётчик.
   Трассы копируются в `reports/dynamic/traces/`.
7. **📊 Создать отчёты покрытия** → `reports/dynamic`
   (`Покрытие_ФО.csv`, `Покрытие_ветвей.csv`, `Сводка_покрытия.csv`); результаты
   покрытия импортируются обратно в `project.db`.

### Выход из проекта
Кнопка **⏏ Выход из проекта** в шапке возвращает к стартовому окну.
При повторном открытии проекта восстанавливаются статус, статистика и доступность
функций.

## Структура проекта на диске

```
<имя-проекта>/
├── project.db              # SQLite: данные анализа, статус, статистика, покрытие
├── reports/
│   ├── static/             # CSV + flowcharts/  (создаются из project.db)
│   └── dynamic/
│       ├── Покрытие_*.csv, Сводка_покрытия.csv
│       └── traces/         # сохранённые трассы
├── orig-sources/           # исходники для инструментации
└── src-instrumented/       # инструментированные исходники
```

## Особенности интерфейса
- **Зоны передачи файлов** работают и перетаскиванием мышью, и кликом ЛКМ
  (проводник). Под администратором drag-and-drop разрешается автоматически (UIPI).
- **Заблокированные элементы** — блеклые; при наведении курсор-запрет и подсказка
  с причиной блокировки.
- **Контраст**: тёмная тема, светлый текст на тёмном фоне и наоборот.
