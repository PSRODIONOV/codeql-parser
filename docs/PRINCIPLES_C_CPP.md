# Принципы статики, динамики и инструментации (C/C++)

Эталонный документ по логике уже реализованного функционала для **C/C++**.
Назначение: (1) зафиксировать принципы и правила, чтобы помнить, почему всё
устроено именно так; (2) служить опорой при доработке остальных языков —
какие принципы переносятся как есть, а что является C/C++-спецификой.

Эталонный тест-проект, на котором всё проверяется:
`examples/test-projects/small-projects/test-project-cpp-branches`
(золотые отчёты — `reports/small-projects/cpp-branches/`, тесты — `tests/test_cpp.py`).

---

## 1. Архитектура: два конвейера

### Статика (двухфазная, поверх `project.db`)
1. **`run_static_analysis`** ([core/project_runner.py](../core/project_runner.py)) —
   выполняет CodeQL-запросы и тяжёлые вычисления (ELK/маршруты/графы) **один раз**
   и складывает сырые наборы + производные данные (+SVG, +статистику) в `project.db`.
   Файлы-отчёты не пишет.
2. **`generate_static_reports`** — быстрый **дамп** из `project.db` в `reports/static`
   (CSV + `flowcharts/`). Повторно CodeQL/ELK не запускаются.

CLI-аналог — [main.py](../main.py): делает то же за один проход (без `project.db`,
сразу в каталог отчётов). **Инвариант: GUI и CLI обязаны давать идентичные отчёты
по одной БД** (см. §8).

### Динамика
`инструментация (копия исходников + датчики)` → `сборка и запуск пользователем`
→ `трассы (.log)` → `отчёты покрытия + сопоставление маршрутов`.

## 2. Извлечение сырых данных (CodeQL-запросы)

Все отчёты строятся из **сырых наборов**, полученных одним батчем запросов
(`run_batch_queries`, БД грузится один раз). Каждый набор — отдельный `.ql`:

| Набор | Запрос | Содержит |
|---|---|---|
| `functional` | `functional_objects.ql` | ФО (функции/методы): имя, файл, строка |
| `info` | `info_objects.ql` | ИО (переменные/параметры/поля) |
| `control` | `control_matrix.ql` | связи по управлению (вызовы) |
| `data` | `data_matrix.ql` | доступ ФО к переменным |
| `arg_flow` / `file_flow` | `arg_flow.ql` / `file_flow.ql` | аргументные потоки / обращения к файлам |
| `signature` | `signature_analysis.ql` | опасные конструкции (ПОК/CWE) |
| `flow` | `function_flow.ql` | управляющие конструкции (для блок-схем/маршрутов/ветвей) + геометрия вставки датчика ветви |
| `catch` | `catch_points.ql` | геометрия вставки датчика catch (try может иметь несколько catch-клауз) |

**Принцип:** геометрия вставки датчиков (вход/выход ФО — `functional_objects.ql`,
ветви — `function_flow.ql`, catch — `catch_points.ql`) считается прямо при сборе
соответствующих наборов на этапе статики. Инструментатор читает геометрию из
отчётов статики (`Перечень_ФО`/`Перечень_ветвей.csv`), без отдельного запроса
к CodeQL-БД.

Общий фильтр файлов проекта во всех запросах — предикат `isProjectFile`:
совпадение с `${PROJECT_PATTERN}`, нужные расширения (`cpp/c/h/hpp/cc/cxx`),
исключение `/usr/include`, `/usr/lib`, `CMakeFiles` и т.п.

## 3. Модель ФО и правило исключения макро-ФО

### Что является ФО
Функция с определением (`hasDefinition()`, есть тело), исключая:
- `operator…` (перегрузки операторов — `getName().indexOf("operator")=0`);
- компиляторно-сгенерированные (`isCompilerGenerated()` — неявные ctor/dtor);
- **целиком синтезированные макросом** (`isInMacroExpansion()`).

### Правило макро-исключения (двухуровневое)
1. **На уровне CodeQL** — `isInMacroExpansion()` отсекает функции, чьё тело —
   часть раскрытия макроса.
2. **Пост-фильтр по факту** ([core/fo_filters.py](../core/fo_filters.py),
   `filter_macro_synthesized_fo`) — для случаев, где CodeQL репортит позицию
   литерального **аргумента** макроса (X-macro, `##`-склейка имени `get_##suffix`),
   и `isInMacroExpansion()` не срабатывает. Проверка: **короткое имя ФО физически
   присутствует на своей строке исходника** (снимок берётся из `src.zip` через
   `read_source_snapshot`, ключ — basename). Перед сравнением из строки
   **вырезаются комментарии** — иначе упоминание имени в комментарии замаскировало
   бы `##`-склейку.

### Каскад исключения (ключевое требование)
> Если ФО исключён — исключаются и его **ветви, и ИО, и опасные конструкции (ПОК)**.

После пометки исключённых имён фильтруются `info`, `data`, `flow` **и `signature`**
(см. `run_static_analysis` и симметрично `main.py`). Исключённый ФО не оставляет
следов ни в одном отчёте. Регресс: `run_macro` (целиком из макроса, содержит
`system()`) — его CWE-078 не попадает в сигнатурный анализ.

**Различай три макро-случая** (`macro_demo.cpp`):
- имя целиком из макроса (`get_answer` через `##`) → **исключается**;
- тело/`{` из макроса, имя настоящее (`macro_body`, паттерн JNI_ENTRY) →
  **инструментируется** (с учётом макро-`{`);
- `void f() MACRO_BODY;` — всё тело один макрос, имя настоящее → инструментируется
  обёрткой.

## 4. Статические отчёты

### 4.1 Сигнатурный анализ (ПОК) — `Сигнатурный_анализ_кода.csv`
Плоский перечень опасных конструкций: CWE, категория, сигнатура, модуль, ФО,
местоположение `file:line`, **фрагмент кода**, источник правила.
Фрагмент = строка исходника из снимка `src.zip` (`source_by_base`, `.strip()`).
**И CLI, и GUI обязаны передавать снимок** — иначе фрагмент остаётся пустым.

### 4.2 Перечень ветвей — `Перечень_ветвей.csv`
Что **считается ветвью** (источник — `function_flow.ql` → перечислитель в
[viz/flowchart_generator.py](../viz/flowchart_generator.py)):

| Тип | Учитывается | Примечание |
|---|---|---|
| `if` / `else` | да | plain else — отдельная ветвь |
| `for` / `while` / `do` | да | |
| `try` / `catch` | да | catch — отдельная ветвь со своим номером, не общим с try |
| `case` / `default` | да | каждая метка — отдельная ветвь |
| range-for | **нет** | негатив |
| тернарный `?:` | **нет** | |
| `&&` / `\|\|` | **нет** | короткое замыкание не ветвь |
| `goto` (сам по себе) | **нет** | но управляющий им `if` — да |
| из макроса | **нет** | `isMacroGeneratedControl` |

**else-if разворачивается в цепочку отдельных `if`** (`else if` = `IfStmt` в позиции
else): нумеруются 1..N как самостоятельные ветви. В `function_flow.ql` else-if
репортится как `else` родителя (нужно перечислителю для деления «да/нет»), но
**фантомная `else`-запись не создаётся** — синтез else пропускается, если `else_line`
совпадает со строкой вложенного `if`.

**switch — N-арное решение**: 7 `case` + 1 `default` → ветви 1..8;
маршруты строятся с учётом fallthrough (рукав = `case[i:]` до `break`).

### 4.3 Блок-схемы
Рендереры: ELK (стандартный/ось), **DRAKON** (упрощённый, без зависимостей).
Режим «упрощённый» заменяет текст узлов типовыми метками (Условие/Цикл/Базовый блок).
DRAKON только рисует SVG; маршруты/ветви он не вычисляет — для них второй проход
через `FlowchartGenerator` без рендеринга (одинаково в CLI и GUI).

### 4.4 Маршруты и графы
Маршруты = базисные пути перечислителя (`_decision_outcomes`/`_arms`/`build_path`),
бинарные решения (if/цикл) и N-арные (switch с fallthrough). Лимит — `max_routes`
на ФО. Графы (функций/ветвей/маршрутов) пересобираются из `control` + `branch_edges`.

## 5. Инструментация (C/C++)

### 5.1 Источник геометрии
Геометрия считается в `functional_objects.ql`/`function_flow.ql`/
`catch_points.ql` и сохраняется в `Перечень_ФО`/`Перечень_ветвей.csv`;
инструментатор (`read_fo_geometry`/`read_branch_geometry` в
`dynamic/_instrument_common.py`) читает её прямо из этих отчётов и
собирает в единый список точек: `kind` (entry/branch), `func`, `file`,
`ref_line`, `ins_line`, `ins_col`, `has_block`, `btype`, `end_line`, `end_col`.
Нумерация ФО/ветвей берётся из **статических отчётов** (`Перечень_ФО`,
`Перечень_ветвей`) → совпадает со статикой 1:1.

### 5.2 Типы датчиков и формат трасс
Текст датчика: `__TRACE_FN(fo, se, sx)` (вход/выход ФО через `cleanup`-атрибут —
ловит все выходы, включая `return` из середины и проброс исключений) и
`__TRACE(sid, fo, br)` (ветвь). В трассу пишутся строки `fo:br`:
- `fo:0` — вход в ФО;
- `fo:-1` — выход из ФО;
- `fo:N` — сработала ветвь №N.

### 5.3 Правила вставки (три режима `has_block`)
- **`has_block=1`** — тело в `{…}`: датчик сразу после `{`.
  ⚠️ Эта `{` может быть **синтезирована макросом** (JNI_ENTRY и т.п.) — её нет
  в литеральном тексте на координате. Инструментатор проверяет символ в файле и,
  если `{` не на месте, использует `end_line/end_col` как запасной якорь.
- **`has_block=0`** — одиночный оператор без скобок: оборачивается
  `{ датчик; оператор; }`.
- **`has_block=2`** — метка `case/default`: текст датчика вставляется **прямо
  после `:`** без обёртки (все метки делят общий блок switch).

Тонкости (отлажены на эталоне):
- **Off-by-one для case**: `ins_col = endColumn+1`, допускается `0 ≤ col ≤ len(ln)+1`
  (двоеточие в конце строки). Иначе датчики case не вставлялись.
- **`refLine` для case** = строка самой метки (не строки `switch`) — иначе все
  метки получали номер первой ветви (`br=1`).
- **Сортировка вставок**: при равной колонке «закрытие раньше открытия»
  (`0 if x=="close" else 1`) — иначе ломались однострочные `if … else` и пустой
  `then` (висячий else).
- **Заголовки не префаются** из дерева (нужны как объявления для сборки).

### 5.4 Рантайм
**Single-header weak-символьный** [`__trace_singlehdr.h`](../dynamic/runtime/__trace_singlehdr.h)
(копируется как `__trace.h`, с префиксом `#define CQ_LANG "<lang>"`):
- кладётся в системный include-путь (`/usr/include`) → виден из любого файла
  независимо от `-I` и числа бинарников;
- реализация целиком в заголовке, помечена `__attribute__((weak))` → линкер сводит
  N копий в одну на бинарник (одно состояние: счётчики, файл трасс, стек маршрутов);
- требует **GCC/Clang** (weak + cleanup); MSVC не компилирует;
- трасса: `$HOME/<lang>-ГГГГММДД-ЧЧММСС-<pid>.log` (на Windows — `%USERPROFILE%`),
  ротация при 100 МБ;
- **анти-спам**: каждый `sid` пишет coverage-строку только на срабатываниях
  1,2,4,8,16… (степени двойки).

Для проектов со своей сборкой — `instrument_c_make.py` (этот же single-header
рантайм); для прямой сборки — `instrument_cpp.py`.

### 5.5 Согласованность
Белый/чёрный список файлов инструментатора — **тот же**, что в статике
(`apply_file_filters`/`set_file_filters`), чтобы оба этапа видели одно подмножество.
Дерево исходников инструментатор берёт из `src.zip` БД (точный снимок того, что
анализировал CodeQL), отдельный каталог не нужен. Финальная проверка —
`g++ -fsyntax-only`; ошибки → `Отчёт_об_ошибках_вставки.csv`.

## 6. Фактические маршруты и сопоставление со статикой

### 6.1 Запись маршрута в рантайме
Стек кадров на вызовы ФО. На входе (`br==0`) — новый кадр (self первым в цепочке
вызовов). На ветви (`br>=1`) — номер дописывается в текущий кадр **только если ФО
ветви == ФО верхнего кадра** (иначе ветвь, сработавшая после снятия кадра — напр.
catch function-try-block — прилипла бы к вызывающему как фантомная `#1` у `main`).
На выходе (`br==-1`) выгружаются две строки:
- `R fo:b1>b2>…` — маршрут по **ветвям**;
- `C fo:fo>c1>c2>…` — цепочка **вызовов**.
Уникальные пути пишутся 1,2,4,8… раз (анти-спам уровня маршрута, FNV-хэш сигнатуры).

### 6.2 Нормализация при матчинге ([route_match_report.py](../dynamic/route_match_report.py))
Чтобы фактический маршрут совпал со статическим базисным:
- **Схлопывание циклов** (`_collapse_cycles`): подряд идущие повторы блоков
  сводятся к одному проходу (`(1,2,1,2)→(1,2)`). Разные ветви на итерациях
  (continue/выход) не схлопываются → останутся «непредусмотренными».
- **Развёртка else-if/switch** — выравнивание с тем, как статика перечисляет ветви.
- `is_instrumented`: датчик ветви стоит на «да»-стороне (then/тело цикла/try-блок);
  для `try/case/default` считается сработавшим всегда (вход при любом исходе/через
  fallthrough).

Типы записей в отчёте: **статический** (отмечается, исполнялся ли),
**непредусмотренный** (реальный путь без статического аналога — комбинированный
прогон, исключения), **неоднозначно (try)** (маршруты, различимые лишь исходом try).
Остаточные «непредусмотренные» (exception-flow try/catch, шаблонные потоки,
varying-iteration циклы) — документированно неустранимы.

## 7. Отчёты покрытия

Из трасс ([coverage_report.py](../dynamic/coverage_report.py)):
- **`Покрытие_ФО.csv`**, **`Покрытие_ветвей.csv`**, **`Сводка_покрытия.csv`** —
  да / нет / не инстр.;
- **`Карта_датчиков.csv`** — какие датчики куда вставлены (№ ФО, `Запись (br)`:
  `0` вход / `-1` выход / `N` ветвь, тип, файл) — для сверки вставки.

Сверка статика↔динамика: каждая ветвь из `Перечень_ветвей` должна иметь
`исполнялся=да` при достаточном прогоне (эталон: 95/95).

## 8. Инварианты согласованности

1. **GUI = CLI**: один код отчётов (`ReportGenerator`, генераторы) на обоих путях.
2. **Статика = инструментация**: нумерация ФО/ветвей из статических отчётов;
   геометрия из `Перечень_ФО`/`Перечень_ветвей.csv`; общий фильтр файлов; общий
   снимок `src.zip`.
3. **Статика = динамика**: `Перечень_ветвей` ↔ `Покрытие_ветвей` ↔ `Карта_датчиков`
   совпадают по составу и нумерации; фактические маршруты сводимы к статическим.
4. **Исключение каскадно**: исключённый ФО исчезает из ФО, ИО, ветвей, ПОК,
   маршрутов и инструментации одновременно.

## 9. Чек-лист переноса на другой язык

### Переносится как есть (язык-независимое ядро)
- Двухфазная статика поверх `project.db`; формат CSV-отчётов; `ReportGenerator`.
- Перечислитель маршрутов/ветвей и графы (`flowchart_generator`, `graph_builder`) —
  работают от набора `flow`, не от языка.
- Модель трасс `fo:0/-1/N`, строки `R`/`C`, анти-спам степенями двойки,
  схлопывание циклов и логика матчинга (`route_match_report.py`).
- Инвариант «нумерация из статических отчётов» и каскад исключения.

### C/C++-специфика (требует языкового аналога)
- **Препроцессор/макросы**: двухуровневое макро-исключение (`isInMacroExpansion`
  + пост-фильтр по строке исходника), макро-`{` и три режима `has_block`. В языках
  без препроцессора (Java/Python/JS) этот слой не нужен — но **нужен свой аналог
  «синтезированных» сущностей** (lambda/синтетические методы, декораторы).
- **Источник исходников** `src.zip` внутри CodeQL-БД и `read_source_snapshot` по
  basename — для PHP/SQL источник иной (каталог), снимок может быть пустым.
- **Single-header weak-рантайм** — чисто C/C++ механизм линковки. Другие языки
  имеют свои рантаймы (`cqtrace.py/js/php`, `Cqtrace.java`) с иной доставкой
  (import/require/require_once/пакет) и иным местом трасс (JS — stdout!).

### Что обязан предоставить каждый новый язык
1. Набор `.ql` (или иной анализатор — Joern для PHP, SqlAnalyzer для SQL),
   дающий те же сырые наборы (`functional/info/control/data/signature/flow`).
2. Определение «что есть ФО» и «что есть ветвь» в терминах языка + правило
   исключения синтетических сущностей (с каскадом).
3. Геометрию точек вставки (аналог `probe`) ИЛИ свой механизм инструментации,
   но с **нумерацией из статических отчётов**.
4. Рантайм, пишущий трассы в формате `fo:0/-1/N` (+ опционально `R`/`C`), и
   документированный способ доставки рантайма и расположения трасс.
5. Регресс по образцу `tests/test_cpp.py` на эталонном тест-проекте с золотыми
   отчётами (определение ветвей, карта датчиков, покрытие, сигнатуры, исключения).
